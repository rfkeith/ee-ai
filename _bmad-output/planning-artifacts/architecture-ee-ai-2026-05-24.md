## 1. System Overview

Engineering Excellence is a single Astro site. **In v1**, it is fully static — generated at build time from markdown article files and Astro page templates, served as plain HTML/CSS/JS from the author's VPS. There is no server-side runtime, no database, no authentication, no API surface, and no client-side state beyond what Astro emits for view-transition / hydration.

**In v1.1** (CONFIRMED 2026-05-26), the site introduces its first SSR posture: Astro hybrid render mode adds server-rendered routes for the course (gated lesson pages, account management, Stripe webhook receiver) while keeping essay pages statically generated. The same VPS hosts both — the static `dist/` continues to be served as before, with a Node SSR process (`@astrojs/node` adapter) handling the gated routes behind the same reverse proxy. v1.1 also introduces a database (managed PostgreSQL) and an auth provider (Better Auth, self-hosted in the Node process).

The system has three concerns:

1. **Content** — markdown articles validated by a Zod schema.
2. **Presentation** — Astro page templates + layouts + Tailwind utility CSS.
3. **Distribution** — static output (`dist/`) deployed to a host.

[NEEDS INPUT] Distribution target is not codified in the repo. Likely candidates: Vercel, Netlify, Cloudflare Pages, GitHub Pages. The `site` field in `astro.config.mjs` is set to `https://engineeringexcellence.dev` (used for canonical URLs and sitemap) but says nothing about the host.

## 4. Content Model

[VERIFIED — `src/content.config.ts`]

A single Astro content collection, `blog`, loaded via `glob({ pattern: '**/*.md', base: './src/content/blog' })`.

```ts
schema: z.object({
  title:        z.string(),
  description:  z.string(),
  pubDate:      z.coerce.date(),
  updatedDate:  z.coerce.date().optional(),
  author:       z.string().default('Robert'),
  category:     z.enum(['leadership', 'best-practices', 'career', 'ai']),
  tags:         z.array(z.string()),
  draft:        z.boolean().default(true),
  sources:      z.array(z.object({ book: z.string(), author: z.string() })).optional(),
  series:       z.object({                  // v1, added 2026-05-24 — see PRD FR-26
                  name:  z.string(),
                  order: z.number().int().positive(),
                }).optional(),
})
```

**Implications:**

- **Default-draft.** Adding a new file with omitted `draft` keeps it hidden. The author must explicitly publish (`draft: false`).
- **Closed category enum.** Adding a fifth topic requires a schema change. Discourages drift, forces decision.
- **Open tag list.** No taxonomy enforcement — tags are free-form and may diverge.
- **Sources captured, not rendered.** Schema accepts source attribution but no component reads it. Forward-compatible — a future "Sources" section is a pure-rendering change with no migration.
- **No author entity.** `author` is a string with a default. Multi-author support would require a schema change and component updates.
- **No series / cross-link relationship.** Article-to-article relationships (e.g., `senior-to-staff-v3` supersedes `v2`) are not modeled.

## 6. Presentation Layer

[VERIFIED — `BaseLayout.astro`, `BlogPostLayout.astro`, `src/styles/global.css`]

- **`BaseLayout`** owns the HTML shell, head metadata (`title | Engineering Excellence`, description, viewport, favicon SVG), and the Header/Main/Footer skeleton. Takes `title` and optional `description` props.
- **`BlogPostLayout`** wraps `BaseLayout` and adds the article-detail header (category chip, h1, description, date, optional updated date, optional reading time, tag list) plus the `prose-purple` body container.
- **Tailwind** is loaded via the Vite plugin (Tailwind v4 — no `tailwind.config.js`). Custom rules live in `src/styles/global.css`. Class composition is utility-first throughout; no CSS modules or component-scoped styles observed.
- **Design tokens** (see UX doc) are encoded as raw Tailwind classes (`purple-800`, `gray-500`, `gray-900`, `gray-200`). There is no theme-token file.

## 8. Cross-Cutting Concerns

[INFERRED] SSG + a small CSS bundle + no client JS framework gives a strong baseline. ## 9. Key Architectural Decisions

| ID | Decision | Rationale | Alternatives considered |
|---|---|---|---|
| AD-1 | Static site generation (Astro 5) | Zero runtime, low cost, fast TTFB, content is read-only and infrequent-change | Next.js (heavier, SSR overkill); Hugo (less ergonomic for the author's stack); MDX-only (no built-in routing) |
| AD-2 | Content collections + Zod schema | Build-time validation; impossible to publish with broken frontmatter | Free-form markdown + runtime parsing (loses safety); CMS (overkill for one author) |
| AD-3 | Tailwind v4 utility-first | Author velocity; design tokens live next to markup; small CSS surface | CSS modules + design tokens (heavier); CSS-in-JS (no runtime, conflicts with SSG ethos) |
| AD-4 | Closed category enum | Forces deliberate taxonomy decisions; categories are URL surface | Free-form categories (taxonomy drift); separate categories+tags both free-form (current is `enum + free tags`, the chosen middle) |
| AD-5 | Markdown-only article authoring | Lowest friction; author already thinks in markdown; renders in any tool | MDX (allows components in articles but adds a learning curve; deferred until needed) |

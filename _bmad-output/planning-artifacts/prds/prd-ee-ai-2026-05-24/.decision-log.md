## 2026-05-26 — Q14 / Q15 / Q16 / Arch Q9 resolved — v1.1 stack locked

**Decisions:**

- **Q14 Auth provider: Better Auth.** Recommended over the alternatives because: self-hosted (matches control bias), TypeScript-native, native Stripe plugin (handles billing-to-entitlement bridge in one place), PostgreSQL adapter via Drizzle, free, active development. Lucia would have been the historic answer but the project announced sunset in 2024; Better Auth is the spiritual successor that the Astro community is consolidating around. Clerk and Supabase Auth rejected because both are managed services that conflict with the control bias and add a third-party dependency the user explicitly wants to avoid.
- **Q15 Entitlement and progress storage: Managed PostgreSQL** (same DB as everything else for the course). Both `entitlements` and `lesson_progress` tables live alongside the Better Auth `users`/`sessions` tables. Stripe customer ID stored as a column on the user row for reconciliation. Rejected alternatives: Stripe customer metadata (too limited, hard to query), auth-provider user metadata (couples entitlement schema to auth provider's schema, and Better Auth's metadata model is intentionally lean).
- **Q16 First database: PostgreSQL.** Default recommendation Neon (serverless tier, no host lock-in, swappable for Supabase/Railway/Render/DO/Hetzner via a connection-string change). Drizzle ORM for access (TypeScript-first, integrates with Better Auth, supports migration discipline a paid product needs).
- **Arch Q9 v1.1 SSR deployment posture: Stay on VPS.** `@astrojs/node` adapter runs the SSR process on the existing VPS behind the existing reverse proxy. Astro hybrid render mode lets essay pages stay statically built (no SSR cost for the 99% of traffic that hits them) while course/account/api routes opt in to SSR. Split and migrate options rejected — both add ops complexity the user explicitly wants to avoid.

**Why these decisions hang together coherently:**

The choices form a single coherent v1.1 stack:
- Better Auth + Drizzle + Postgres + `@astrojs/node` + Stripe — all run in one Node process on the existing VPS.
- No third-party SaaS in the critical path. Postgres is managed (Neon or equivalent) but swappable; everything else is self-hosted.
- The VPS continues to be the single deployment target. The build script (NFR-6 venue) extends to also build and restart the Node SSR process; the static `dist/` continues to be served as before.
- Stripe is the only external API in the critical path, and the webhook handler is in the SSR routes (no separate service).

- **PRD §9 Q14, Q15, Q16** all RESOLVED in place with the chosen options and rejected alternatives named.
- **PRD §4.7 stack-additions block** rewritten with the confirmed components and their integration roles.

## 2026-05-26 — Critical #1 closed: FR-21 cookie consent hard-gated, hand-rolled

**Decision: Cookie consent is hand-rolled (no third-party CMP). FR-21 hardened with explicit pre-consent posture, mechanism, and two Playwright tests added to the NFR-6 smoke gate to make the gate verifiable.**

- *Why hand-rolled (over Cookiebot / Klaro / OSS CMPs):* Gate logic stays in one file the author can audit. No $10-30/month vendor cost. No third-party load dependency (avoids the irony of a "consent" script that itself sets cookies). Implementation cost is modest (~80 lines of Astro + inline JS).
- *What "hard-gated" means now in the FR (these are non-negotiable):*
  - No GA4 script tag in the document before consent — not async-loaded with a consent check after, not "default deny" within Consent Mode v2 — *not present at all*.
  - No GA4 cookies (`_ga`, `_ga_<id>`, etc.) set before consent.
  - No network requests to `google-analytics.com`, `googletagmanager.com`, `analytics.google.com`, or `stats.g.doubleclick.net` before consent.
  - The only first-party cookie pre-consent is the `ee-consent` cookie itself.
- *Mechanism:* `ee-consent` cookie (SameSite=Lax, 6mo expiry). Server-side per-page render checks the cookie and decides whether to emit the GA4 snippet at all. Client-side on Accept: set cookie + inject snippet runtime. On Reject: set cookie, do nothing further. Re-consent via `/privacy/` "Change cookie preferences" affordance.
- *Banner UX:* Bottom-of-viewport, sticky, two equal-weight buttons ("Accept analytics" / "Reject"), one link to `/privacy/`. No dark patterns.
- *Verification baked into NFR-6:* Two Playwright tests — (a) clean cookie jar load asserts no GA-domain requests + no GA cookies; (b) `ee-consent=accept` preset asserts GA4 loads and `collect` fires. This makes the hard-gate a build invariant, not an intention.
- *Validation impact:* Critical #1 from the adversarial review is now closed. Re-running validation would drop one critical → 2 remaining (v1.1 scope realism, v1 scope sprawl).

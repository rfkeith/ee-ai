## 0. Document Purpose

This PRD describes Engineering Excellence, a static blog at engineeringexcellence.dev publishing research-backed essays on software engineering leadership, best practices, career progression, and AI in enterprise settings. It is reverse-engineered from the live codebase to provide grounding context for subsequent BMad workflow runs (architecture, epics, stories) on this project.

## 1. Vision

[VERIFIED — from `src/pages/index.astro` tagline and `src/pages/about.astro` copy]

Engineering Excellence is a personal essay site for senior engineers, tech leads, and engineering managers working inside enterprise organizations. Each essay takes a piece of conventional engineering wisdom, names what's incomplete or wrong about it in practice, and provides a more grounded alternative supported by named sources (typically books on leadership, organizational behavior, or career strategy).

[INFERRED — from article style, sources schema, voice in `shielding-vs-exposure.md` and `senior-to-staff-v3.md`, and author direction]

AI tools now write most of the code. That changes what makes engineers valuable at every level.

For experienced engineers, the new skill is telling the AI what to build and catching its mistakes. In a big-company setting, where "almost right" code causes real harm, that skill is rare.

For junior engineers, the challenge is different. They need to learn what good code looks like in real enterprise systems — taste, design judgment, the habits of craft — without the years of slow practice that used to build those skills. AI tools can short-circuit the typing. They can't short-circuit the judgment.

Engineering Excellence serves both groups along distinct paths:

- **A free essay track for experienced engineers** — short, honest takes on craft, leadership, career, and what AI actually changes.
- **A paid self-paced course for junior engineers** — a structured curriculum that walks them from "can write code with AI help" to "can build enterprise-grade software." Currently post-v1; see §4.7.

The voice across both paths is honest, not preachy. Most engineering advice you read is half right. Articles and lessons try to give you the other half. Every claim points to real sources — usually books — so you can check the thinking for yourself.

The audience-building infrastructure (newsletter, RSS, sitemap, search) exists to compound the work, not to substitute for it. Success metrics should weight "did we ship 1-2 articles/month for 12 months" higher than any traffic number.

[CONFIRMED — author affirmed 2026-05-24]

The site is run as a long-term audience and reputation platform, with runway toward future products beyond the course (books, talks, possibly more courses). The paid course (§4.7) is the first monetized product, not the central purpose. The essay track is the durable asset; the course is leverage off it. This means audience-building work — newsletter, RSS, sitemap, SEO hygiene, cadence discipline — sits alongside the course in the roadmap, not after it.

## 2. Target User

The site serves two distinct readers along two distinct content paths. Both face new versions of the same question — what makes engineering work valuable when AI writes most of the code? — but the answers differ by career stage, so the content shape and the business model differ too.

[INFERRED] Mid-career ICs and first-line leaders inside large engineering organizations (FAANG-adjacent or enterprise IT). They already read engineering blogs and are skeptical of pithy advice. They read on a phone or laptop, mostly during work hours. **Free.**

#### 2.1.2 Junior engineer (paid course track) — POST-v1

[INFERRED — from author direction. v1.1+ scope marker added 2026-05-24 during validation.]

### 2.2 Jobs To Be Done

AI now writes most of the code. Both readers face new versions of an old question: what makes my work valuable? The two groups need different answers, so the site serves them through different jobs.

#### A. Jobs — experienced engineers (free essay track)

AI now writes most of the code. The engineer's job is moving up a level — deciding what to build, telling the AI how to build it, and catching the bugs that pass tests but still cause problems. The essay track helps with that work.

**Focus on the whole system, not single lines.**
- *Tell the AI what good looks like.* The engineer has to know what "good" means for this project — and say it clearly enough that the AI doesn't go off track.
- *Spot AI mistakes fast.* AI code usually compiles and looks normal. The real problems are subtler: shaky design, slow code paths, error handling that hides real bugs. Readers need quick ways to catch these.
- *Keep the codebase from falling apart.* AI tools mean more code lands every day. Without care, the design rots fast. Readers need ways to keep things clean without slowing the team down.

**Stay valuable as the work changes.**
- *Cut through hype and doom.* Some people say AI will replace all engineers. Others say real engineers don't use AI. Both are wrong. Readers want honest takes on what has actually changed and what hasn't.
- *Build the skills that matter now.* Typing fast doesn't matter much anymore. What matters: breaking down problems, having taste, reviewing code well, designing systems, and getting good work out of unpredictable teammates — both human and AI.
- *Stay employed through the shift.* Engineers who took pride in their craft are asking "what counts as good now?" They want grounded answers.

**Get work done inside big companies.**
- *Work in real codebases, not toy ones.* Most engineering writing assumes a fresh start. Readers deal with old code, stakeholders, and compliance rules. They want advice that fits their reality.
- *Lead a team that uses AI well.* Tech leads and managers have to set rules: how to review code, when to use AI, when to slow down, when to ship. They want examples that worked and warnings about what didn't.
- *Design systems that handle AI-written code.* When half the code comes from AI, how should you draw module boundaries? How do you test? Who owns what? Readers want answers built for this new reality.

**Back up arguments with real sources.**
- *Cite books, not tweets.* When the reader pushes back on a stakeholder, defends a design, or coaches a junior, they want to point at a real source — not something they half-remember from social media.

**Navigate career transitions.**
- *Move through hard career steps.* Senior to Staff. IC to manager. Coder to system steward. These shifts are uncomfortable, and most writing about them is either preachy or jaded. Readers want honest accounts.

#### B. Jobs — junior engineers (paid course track) — POST-v1

*v1.1+ scope. Drives §4.7 only. v1 has no surface that serves this group.*

Junior engineers can produce code with AI help on day one. The hard part is judgment — knowing what to build, why one approach beats another, what's wrong with code that "works," and how to operate inside a real engineering team. AI can short-circuit typing. It can't short-circuit taste. The course teaches the judgment side.

**Learn what "good" actually looks like.**
- *See enterprise code, not toy examples.* Most tutorials use one-file demos. Real code lives in big systems with constraints. The course shows the difference.
- *Build taste, not just output.* Why one design beats another. Why one PR is reviewable and another isn't. Why one error message helps and another doesn't.
- *Read a real codebase well.* Most junior engineers can write code faster than they can understand a 50,000-line system. The course teaches the reading skill.

**Use AI as an accelerator, not a crutch.**
- *Direct the AI without losing the underlying skill.* Use the agent to type — but understand what it's typing well enough to catch its mistakes.
- *Avoid the dependence trap.* Engineers who skip the fundamentals can't fix what the AI gets wrong. The course builds those fundamentals alongside the AI workflow.

**Build the habits enterprise expects.**
- *Test, review, debug, design.* The four habits that separate junior from senior. The course walks each one.
- *Communicate engineering tradeoffs.* Write design docs. Defend a choice in a review. Explain a bug to a non-engineer. These are the multipliers.

**Cut years off the timeline.**
- *Follow a structured path, not a content feed.* The course is a curriculum, not a YouTube playlist. Pick it up, work through it, finish with the skills it promises.
- *Get value worth paying for.* The course exists to be bought. It has to deliver enough that a buyer would recommend it.

## 3. Glossary

- **Article** — A single markdown file under `src/content/blog/{category}/*.md` validated by the Zod schema in `src/content.config.ts`. Articles have exactly one category, zero-or-more tags, and zero-or-more sources.
- **Category** — One of `leadership | best-practices | career | ai` (enforced by Zod enum). Determines the article's URL prefix in browsing but not in canonical URL (canonical is `/blog/{slug}/`).
- **Tag** — Free-form string in `tags: []`. Used for cross-cutting topics that don't fit the category taxonomy.
- **Source** — A `{book, author}` reference cited in an article's frontmatter. Allows downstream features (citation pages, "books that informed this piece") but currently rendered nowhere in v1.
- **Draft** — `draft: true` in frontmatter excludes the article from the published collection (the index/category/tag pages filter on `!data.draft`).
- **Reader** — Anonymous, unauthenticated visitor to the site. Reads free essays. No account.
- **Free track** — All non-gated content on the site: essays, about, blog index, category and tag pages. Default audience: experienced engineers.
- **Paid track** — The monetized course content under §4.7. Default audience: junior engineers.
- **Course** — The single monetized product (v1.1). Self-paced. One course at launch; future courses possible but out of scope here.
- **Lesson** — One unit of course content. Sequential within the course.
- **Learner** — A paying customer with access to course content. Has an account.

**Functional Requirements:**

**Consequences (testable):**

#### FR-2: Display reading time

[CONFIRMED 2026-05-24]

Reader sees an estimated reading time on each article.

**Consequences (testable):**
- Reading time computed at build time from the article body word count (baseline ~250 words/minute).
- Implementation: a remark plugin (`remark-reading-time` or equivalent) injects the value into article frontmatter / `readingTime` prop.
- Article header shows `· {n} min read`.

- Value is static per build — no client-side recompute.

#### FR-25: Render source citations at article footer

[CONFIRMED 2026-05-24]

Reader sees the article's source citations rendered as a footer block at the end of the article body.

**Consequences (testable):**
- For articles with `sources: [{book, author}, ...]` populated, a "Sources cited in this piece" block renders after the prose body and before any future end-of-article surfaces.

#### FR-28: Related articles at article footer (algorithmic — tag overlap) — v1.05

[CONFIRMED 2026-05-24; deferred to v1.05 2026-05-26 — see §6.1.1]

Reader sees up to 3 related articles at the end of each article, computed from tag overlap with the current piece.

**Consequences (testable):**
- Algorithm: for each other published article, count how many tags it shares with the current article. Rank descending. Tie-breaker: more recent `pubDate`. Take the top 3.
- Computed at build time per article — no runtime cost.
- Block renders after FR-25 source footer.
- If no other article shares ≥1 tag, the block does not render (no "Related Articles" header with empty list).
- Visual treatment: same `BlogPostCard` component used on the home and blog index — consistency reinforces the site's design language.
- Each card shows title, description, category chip, date, tags.

**Notes:**

- **Coupled to tag hygiene (NFR-7).** Bad tags directly produce bad related-article results. The `npm run tags` helper script is doubly important now — drift here is visible in every article footer.
- **Build-time near-duplicate-tag check (CONFIRMED 2026-05-24 — added during validation).** The build emits a warning when two tags differ only by Levenshtein distance ≤ 2, or by case/hyphenation/singular-plural pairs (e.g., `team-management` vs `team management` vs `teams`). Warning, not failure — author resolves by consolidating or by explicitly allowlisting the pair. Implementation: ~15 lines in `astro.config.mjs` integration hook or a content-collection refine. Closes the FR-28 ↔ NFR-7 coupling risk.
- **Cold-start caveat.** At low article counts (current: 5), most articles will surface 1–2 related, not 3. Acceptable; the block just degrades gracefully.
- **Future option:** A small "Why these?" affordance (hover/tap to see the matched tags) could explain the algorithm to readers. Not in scope for v1.

#### FR-27: Render series navigation in article — v1.05

[CONFIRMED 2026-05-24; deferred to v1.05 2026-05-26 — see §6.1.1]

Reader of an article that belongs to a series sees a series ribbon with the series name, the part number, and links to the previous and next parts.

**Consequences (testable):**
- For articles with `series: {name, order}` populated, a "Part {order} of {name}" ribbon renders **beneath the byline, above the tag list**, in the article header. (UX doc revision may override; v1 implementation uses this placement.)
- Previous and next links resolve to articles with the same `series.name` and `order - 1` / `order + 1` respectively.
- If no adjacent part exists at build time, the corresponding link is omitted (no broken links).
- For articles without a `series` field, the ribbon does not render.
- A standalone `/series/{slug}/` index page is **not** included in v1 — defer until at least two series exist on the site to validate the IA.

### 4.6 Versioned drafts (AI-assisted authoring)

[CONFIRMED — author affirmed 2026-05-24]

**Description:** The author uses AI to draft content, which produces multiple variations of the same piece during iteration. Each variation lives as its own `.md` file with a version suffix (`-v2`, `-v3`). The `draft: true/false` flag is the sole publication gate — only the chosen variation gets `draft: false`. Relationships between variations live in the author's head and in filenames, not in the schema.

**Functional Requirements:**

#### FR-12: Coexistence of versioned drafts in the filesystem

Author can keep multiple `.md` files representing variations of the same article. Only one (or zero) is published at a time via `draft: false`. The schema deliberately does not model "v3 supersedes v2" — adding such a relationship would create friction in the drafting loop without serving any reader use case.

**Consequences (testable):**
- Multiple files in the same category folder with version suffixes do not cause build failures — each is treated as an independent article from the loader's perspective.
- Only files with `draft: false` appear in any public surface (FR-3, FR-4, FR-5, FR-6).
- No "related versions" or "previous version" affordance exists in the UI.
- **Slug-collision lint (CONFIRMED 2026-05-26 — promoted to v1).** Build-time check fails when two non-draft articles share a base slug after stripping `-v\d+` suffix. Catches the case where two AI-drafted variations (`foo.md` and `foo-v3.md`) both have `draft: false`. ~10 lines, runs as part of NFR-6. ### 4.8 Analytics & Privacy

[CONFIRMED 2026-05-24]

**Description:** Site uses Google Analytics 4 for traffic and engagement signal. Because GA4 sets cookies and processes behavioral data, the site also needs a cookie consent banner and a privacy policy page. These three pieces ship together — it's not legally clean to deploy GA4 without the other two in most jurisdictions where readers will land (EU, UK, California). Realizes audience-platform measurement (§1, §8).

**Functional Requirements:**

#### FR-20: GA4 integration

Site loads the GA4 measurement script and reports page views and standard engagement signals to Google Analytics — but only after consent (FR-21). The default-deny posture is non-negotiable.

**Consequences (testable):**
- GA4 measurement ID configurable via build-time env var, never hardcoded.
- **GA4 script tag is conditionally rendered**, not async-loaded with a consent check after the fact. If consent is not present, the script tag is not in the document at all — no `<script src="googletagmanager.com/...">`, no inline `gtag()` call, nothing that produces a network request.
- On consent grant, the GA4 snippet is injected into the page (page-reload OR runtime injection via a small in-page script; either works). From that point on, page views and standard engagement events (FR-29) report to GA4.
- Measurement ID injection happens server-side at render time when the consent cookie is present — keeps the gating decision in one place (the page handler), not split between server and client.

#### FR-21: Cookie consent banner — hand-rolled, hard-gated

[CONFIRMED 2026-05-26 — hardened to close validation critical #1]

First-time visitor sees a cookie consent banner. The banner is **hand-rolled** (Astro component + small inline `<script>`); no third-party CMP, no Cookiebot, no Klaro. The hand-rolled choice is deliberate: it keeps the gate logic in one file the author can audit line by line, avoids paying $10–30/month for a third-party script that itself sets cookies, and removes a third-party load dependency.

**Pre-consent posture (non-negotiable — these are the testable hard-gate requirements):**

- **No GA4 script tag** in `<head>` or anywhere in the document until consent is granted.
- **No GA4 cookies** (`_ga`, `_ga_<id>`, `_gid`, `_gat*`) are set before consent.
- **No network requests** to any of: `google-analytics.com`, `googletagmanager.com`, `analytics.google.com`, `stats.g.doubleclick.net` before consent.
- **No essential cookies beyond the consent state itself.** The site currently has no auth, no preferences, no shopping cart — so the consent cookie is the only first-party cookie pre-consent.

**Consent state mechanics (CONFIRMED 2026-05-26, hardened during re-validation fix to resolve static-vs-SSR contradiction):**

- Stored in a first-party cookie named `ee-consent` with value `accept` or `reject`. SameSite=Lax. Path=/. 6-month expiry. Set at the time the user clicks the banner button. Not HttpOnly — the client needs to read it on every page load (essay pages are static).
- **Two distinct mechanisms, one shared cookie:**

    **(a) Static essay pages (FR-1–FR-9, the 99% of traffic).** Astro prerenders these — the served HTML is identical for every visitor regardless of cookie state. Therefore: the GA4 snippet is **never** in the served HTML for essay pages. A small inline `<script>` on every page reads `document.cookie`, and IF AND ONLY IF `ee-consent=accept`, dynamically injects the GA4 `<script src="...">` tag into the document at runtime. If the cookie is `reject` or absent, the inline script does nothing. The banner (also rendered statically) handles its own first-visit show/hide via the same cookie check.

    **(b) SSR routes (course, account, api — v1.1+).** Per-request server-side render. The Node route handler reads the `ee-consent` cookie from the request and conditionally renders the GA4 snippet into the response HTML. No client-side injection needed for these routes.

- Both mechanisms produce the same observable outcome (no GA4 script tag, no GA4 cookies, no GA4 network requests pre-consent). The Playwright test suite below verifies the outcome, not the mechanism.

**Banner UX:**

- Renders at the bottom of the viewport on first visit (no prior `ee-consent` cookie present). Sticky; does not cover the article content.
- Two primary buttons: "Accept analytics" and "Reject". Equal visual weight (no dark-pattern emphasis on accept).
- One link: "What this means" → `/privacy/` (FR-22).
- Banner does not re-appear on subsequent visits once a choice exists.

**Re-consent path:**

- `/privacy/` (FR-22) includes a "Change cookie preferences" affordance that clears the `ee-consent` cookie and re-renders the banner on the next page load. No third-party UI required.

**Verification (testable in NFR-6 smoke gate):**

**Test 1 — pre-consent posture (clean-jar load).** Loads `/` with a clean cookie jar (no `ee-consent` cookie) and asserts:
1. The banner is visible.
2. No network requests fire to any of the GA-domain hosts listed above.
3. No GA4 cookies appear in the cookie jar.
4. The DOM does not contain any `<script>` referencing `googletagmanager.com` or `google-analytics.com` (catches the "we shipped the tag but flipped a flag" failure mode).

**Test 2 — post-consent injection on a static page.** Loads `/` with `ee-consent=accept` cookie preset, asserts the GA4 script is dynamically injected into the DOM by the inline script, and that a `collect` request fires to `google-analytics.com`. Specifically tests the static-page mechanism (a).

**Test 3 — post-consent server render on an SSR page (v1.1+, when course routes exist).** Loads `/course/` with `ee-consent=accept` cookie preset, asserts the GA4 snippet is server-rendered into the response HTML (visible in the initial document, not injected client-side). Specifically tests the SSR-route mechanism (b).

These three tests together make the hard-gate posture a build-blocking invariant across both render modes, not a hope. Test 3 ships when the first SSR route lands; Tests 1 and 2 ship with v1.

#### FR-29: Engagement instrumentation for §8 metrics

[CONFIRMED 2026-05-24 — added during validation]

Site collects enough engagement signal to make the §8 success metrics actually measurable. Without this, "median time-on-page ≥ 3 min" is unreachable (GA4's default time-on-page is broken for last-page-in-session) and "external backlinks" has no instrumentation at all.

**Consequences (testable):**
- **GA4 enhanced measurement enabled** for: page views (default), scroll events (default), file downloads (default if present), site search (yes — instrument the FR-23 modal to fire a `search` event with the query). Outbound clicks event enabled (default).
- **Engaged-time event configured** — GA4's `engagement_time_msec` event fires reliably for the §8 "median time-on-page" metric. Default GA4 reports use this, not the broken time-on-page calculation.
- **Google Search Console (GSC) integration** for backlink and search-query signal. v1 setup: verify domain ownership via DNS TXT record, link GSC property to the GA4 property, no in-app integration needed beyond that.
- **No additional cookies** beyond what FR-20 already requires; FR-21 consent gate still governs all of the above.

**Notes:**

- GSC adds no scope to the codebase — it's an external dashboard configured once.

Smoke set at minimum: home, blog index, each of the four category pages, and one article. **Venue: the VPS build script itself.** The script runs `astro build`, then `npx playwright test`, then rsyncs `dist/` to the web root — and aborts before rsync if either step exits non-zero. No `--no-verify`-style override flag. No "I'll just push the dist this time" path. The discipline IS the venue.
- **NFR-8. v1.1 operational posture** (CONFIRMED 2026-05-26 — added during re-validation fix to capture risks of the on-domain stack).

  The v1.1 stack introduces six new-to-author dependencies (Better Auth, Drizzle, managed Postgres, `@astrojs/node`, Stripe webhooks, systemd) into a single Node process on the existing VPS. Failure modes the v1 static deployment didn't have:

  - **Uptime target: 99% for course routes** (~7.2h/month downtime budget — honest for a solo-author SaaS-shape product). Essay routes inherit the v1 static-deploy uptime, which is effectively limited only by the reverse proxy and VPS.
  - **Restart-on-crash:** the Node SSR process runs under systemd with `Restart=on-failure` and a 5-second `RestartSec`. Crash loops cap at 5 restarts in 60s before systemd backs off.
  - **Health-check route:** `/api/health` (SSR) returns 200 + a JSON body confirming DB connectivity (one `SELECT 1` via Drizzle). Reverse proxy may use this for routing decisions; external uptime monitor (Uptime Robot, BetterStack, etc.) pings it every 5 minutes.
  - **Essay-route independence:** if the Node SSR process is down, essay routes (FR-1–FR-9) must keep serving from the static `dist/`. This is a reverse-proxy configuration concern — `/`, `/blog/*`, `/about/`, `/privacy/`, `/tags/*` route to the static files directly; `/course/*`, `/account/*`, `/api/*`, `/login/*` route through Node. Document the proxy config in `DEPLOY.md`.
  - **Degraded modes named explicitly:**
    - DB unreachable → SSR routes return 503 with a static "Service temporarily unavailable" page (no Postgres connection means no entitlement check means no lesson body).
    - Better Auth breaking change in a minor version → version pinned in `package.json`; bump only deliberately, with auth-flow Playwright tests passing first.
    - Stripe webhook outage → existing entitlements unaffected; new purchases queue in Stripe's retry buffer (FR-16b's idempotency posture).
    - Neon throttles free-tier → migration story to paid Neon tier (or to a different managed Postgres) is a connection-string change. No code changes.
  - **Bus-factor risk acknowledged.** Single VPS hosts essays + SSR + DB connections. Acceptable at the v1.1 sales volume implied by FR-16b (~100 sales/month). Revisit when traffic warrants — that means when GA4 (FR-29) shows sustained pressure on either the static-file serving or the Node process resource usage.

Tag-list helper script.** [CONFIRMED 2026-05-24] An `npm run tags` script dumps the existing tag list across all published articles with usage counts (e.g., `leadership (3), team-management (2), engineering-culture (1)`).
